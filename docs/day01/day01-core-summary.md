\# Day 1：项目整体认识与 BFF 调用链



\## 1. 今日目标



Day 1 的目标不是记住所有代码，而是先建立项目的整体认知，回答三个问题：



1\. 这个项目主要负责什么？

2\. 页面请求如何进入后端并找到具体业务方法？

3\. BFF 层为什么不是普通的单表 CRUD？



\---



\## 2. 项目定位



该项目面向酒旅商品与订单业务，主要支撑：



\* 商品搜索、筛选、排序与详情展示

\* 后台订单列表、汇总、详情与导出

\* 多个后端领域服务之间的查询编排

\* 为前端组装可以直接使用的数据结构



项目中的 `trade-bff` 位于前端页面和订单、商品、用户、履约等下游服务之间，主要承担参数转换、服务调用、数据聚合和结果组装。



\---



\## 3. 统一入口：`biz/index.ts`



`biz/index.ts` 是整个业务云函数的统一入口，但它不是具体业务的实现位置。



它主要负责：



1\. 接收页面或外部系统请求

2\. 读取请求中的路由信息

3\. 根据路由信息动态加载业务服务

4\. 执行具体业务方法

5\. 记录日志、耗时和异常

6\. 包装响应并返回调用方



可以将它理解为后端系统的“总前台”或“路由分发器”。



\---



\## 4. 动态路由协议



项目通过三个字段确定最终执行的方法：



```text

moduleName + serviceName + funcName

```



例如：



```text

moduleName  = trade-bff

serviceName = web

funcName    = queryOrderForParty

```



它表示：



1\. 进入 `trade-bff` 模块

2\. 加载 `web` 服务

3\. 执行 `queryOrderForParty` 方法



各字段的职责是：



\* `moduleName`：定位业务模块

\* `serviceName`：定位模块中的服务文件

\* `funcName`：定位服务文件中的具体函数

\* `param`：真正的业务查询参数



这套设计让多个业务接口可以共用同一个云函数入口。



\---



\## 5. BFF 是什么



BFF 的全称是 Backend For Frontend。



在该项目中，调用关系可以理解为：



```text

前端页面

&#x20; ↓

biz 云函数统一入口

&#x20; ↓

trade-bff

&#x20; ↓

订单 / 商品 / 用户 / 履约等下游服务

```



BFF 的主要职责包括：



\* 把页面参数转换为底层服务能够识别的参数

\* 编排多个后端领域服务

\* 并发执行互不依赖的查询

\* 补充页面需要的跨域数据

\* 把底层复杂结构组装为页面模型



一句话总结：



> BFF 是前端页面和多个后端领域服务之间的适配与聚合层。



\---



\## 6. `queryOrderForParty` 核心链路



以一日聚订单列表为例，页面可能传入：



\* 商品名称

\* 用户昵称

\* 联系人信息

\* 订单状态

\* 时间范围

\* 分页参数



但是订单服务更适合使用：



\* `itemId`

\* `unionId`

\* `orderId`

\* `status`

\* `createdAtRange`



因此，`queryOrderForParty` 并不是简单查询一次数据库，而是完成一条完整的查询编排链路。



\### 6.1 参数转换



```text

商品名称

&#x20; ↓ 查询商品服务

itemId

```



```text

用户昵称 / 真实姓名 / 手机号

&#x20; ↓ 查询用户服务

unionId

```



转换完成后，再把 `itemIdList`、`unionIdList` 等条件交给订单查询服务。



\### 6.2 并发查询列表和总数



分页页面同时需要：



\* 当前页订单列表

\* 符合条件的订单总数



代码通过 `Promise.all` 同时执行这两个查询：



```text

查询订单列表 ─┐

&#x20;             ├─ 并发执行

查询订单总数 ─┘

```



因为两个查询互不依赖，并发执行可以减少接口整体等待时间。



\### 6.3 补充跨域信息



获取订单列表后，BFF 还会补充页面需要的数据，例如：



\* 用户信息

\* 私人管家信息

\* 差额信息

\* 子单信息

\* 主订单标记

\* 合并订单标记



这些数据不一定全部属于订单域，因此需要继续调用其他服务。



\### 6.4 组装页面模型



底层服务通常按照各自领域返回数据，而前端需要的是可以直接展示的结构。



BFF 会将数据整理为类似：



```json

{

&#x20; "orderId": "脱敏订单ID",

&#x20; "itemName": "商品名称",

&#x20; "contactName": "联系人",

&#x20; "status": "订单状态",

&#x20; "salePrice": 0,

&#x20; "hasRelatedOrder": false

}

```



最后返回统一分页结构：



```json

{

&#x20; "list": \[],

&#x20; "total": 0

}

```

\## 7. 为什么它比普通 CRUD 更有含金量



普通 CRUD 的常见链路是：



```text

接收参数

→ 查询一张表

→ 返回结果

```



该项目中的 BFF 查询链路是：



```text

接收页面条件

→ 参数转换

→ 跨服务预查询

→ 查询订单列表与总数

→ 补充多个领域的数据

→ 组装页面模型

→ 返回统一分页结构

```



它体现的技术能力包括：



\* 动态路由

\* 参数适配

\* 服务编排

\* 并发查询

\* 跨域数据聚合

\* 页面模型组装

\* 统一日志和异常处理

\## 8. Day 1 核心结论



1\. `biz/index.ts` 是统一入口，不是具体订单业务实现。

2\. `moduleName + serviceName + funcName` 构成动态路由地址。

3\. `trade-bff` 是前端与多个下游领域服务之间的聚合层。

4\. `queryOrderForParty` 的核心不是单纯查库，而是参数转换、跨域查询、并发执行、数据补充和结果组装。

5\. 该项目相较普通 CRUD 更有价值的地方，是它体现了真实业务中的 BFF 编排和跨服务聚合能力。

\## 9. 面试口述版本



这个项目采用统一的云函数入口。页面请求进入 `biz` 云函数后，入口根据 `moduleName`、`serviceName` 和 `funcName` 动态加载对应业务服务并执行具体方法。



以一日聚订单列表为例，请求最终进入 `trade-bff` 的 `queryOrderForParty`。该方法先把商品名称、用户昵称等页面条件转换为 `itemId` 和 `unionId`，再并发查询订单列表与总数，之后补充用户、管家、差额和子单等跨域数据，最后组装成 `list + total` 返回前端。



它的核心价值是 BFF 层的参数适配、服务编排和数据聚合，而不是简单的单表 CRUD。

\## 10. 自测问题



\### 1. `biz/index.ts` 会直接查询订单数据库吗？



不会。它负责接收、解析和分发请求，具体业务由对应的 Service 方法执行。



\### 2. 为什么要把商品名称转换成 `itemId`？



因为商品名称属于页面查询条件，而订单域更适合使用稳定的商品 ID 进行查询。



\### 3. 为什么列表和总数要并发查询？



两者互不依赖，并发执行可以减少接口整体等待时间。



\### 4. 为什么需要 BFF 组装结果？



因为底层服务按照各自领域返回数据，前端需要的是统一、可以直接渲染的页面模型。
