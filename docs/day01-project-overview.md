\## 项目背景



原有电商系统和酒旅系统相互独立。



私域直播间售卖酒旅商品时，需要先创建电商虚拟商品。用户支付后，还可能需要退掉电商订单，再前往酒旅小程序重新下单，导致交易和履约流程割裂。



项目目标是打通直播间、酒旅交易系统、后台和小程序，使酒旅商品能够支持定金与尾款交易，并复用原有报单、履约和出团管理能力。



\## 系统中的 BFF



BFF 位于前端页面和下游领域服务之间，主要负责：



\- 参数转换

\- 多服务调用编排

\- 返回字段组装

\- 多端展示规则适配

\- 统一异常和日志处理



\## 统一路由入口



云函数入口通过三个字段确定实际调用方法：



\- moduleName：业务模块

\- serviceName：模块中的服务

\- funcName：具体执行函数



调用链可以概括为：



页面请求

→ biz 云函数入口

→ moduleName / serviceName / funcName 动态路由

→ BFF 服务方法

→ 下游服务或数据库

→ 聚合返回结果



\## 当前掌握的核心链路



\- 商品搜索：commodity / search / searchItem

\- 一日聚订单列表：trade-bff / web / queryOrderForParty

\- 一日聚汇总：trade-bff / web / querySummaryInfoForParty

\- 订单详情：trade-bff / web / getOrderById

\- 分片导出：trade-bff / web / batchDownloadTradeOrderRawData



\## 今日总结



这个项目不是单一 CRUD，而是一个围绕酒旅商品和订单的 BFF 聚合项目。



其核心价值包括：



\- 统一复杂查询协议

\- 编排多个下游服务

\- 保证多端数据口径一致

\- 支撑运营查询、详情和批量导出
