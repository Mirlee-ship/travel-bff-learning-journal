Day13 面试笔记：用户上下文与权限边界

一、30 秒版本



Day13 我复盘的是项目中的用户身份上下文和权限边界。统一入口识别当前调用者，将 openId、unionId 或后台员工身份整理到 UserContext，再把 param + userContext 一起传给业务方法。param 决定查什么，userContext 决定谁在查以及能查哪些数据。请求结束后会销毁上下文，避免云函数实例复用造成身份污染。



二、60 秒版本



项目中的业务方法不只接收页面参数，还会接收 userContext。



页面参数表达用户想查询的状态、时间或订单 ID，用户上下文则保存当前调用者的身份、门店和权限范围。



后台订单查询不能完全相信前端传入的门店和角色信息，真正的数据范围需要由后端根据可信登录身份判断。



项目中还能看到 OPENID、UNIONID 和内部调用 token。用户身份代表谁在操作，内部 token 代表 BFF 是否有资格调用下游服务，这两类身份不能混在一起。



三、90 秒版本



Day13 我重点理解的是请求身份在 BFF 中的传递和权限边界。



请求进入 biz 统一入口后，系统会获取微信上下文或后台登录身份，并构造当前请求的 UserContext。业务方法的调用可以理解为 service\[funcName](param, userContext)。



param 表示这次请求想做什么，userContext 表示是谁在做、属于哪个门店以及具有什么权限。



例如订单详情不能只根据 tradeOrderId 返回数据，还需要判断当前调用者是否有权查看这笔订单。前端隐藏按钮不能替代后端权限校验，因为请求可以被手动构造。



项目还会从 Redis 读取内部服务 token，用于 BFF 调用下游服务。用户身份和服务身份需要分开管理。请求结束后在 finally 中清理 UserContext，防止云函数实例复用导致身份残留。



我的理解重点是身份上下文、权限范围和安全边界，不是公司统一认证平台的底层设计。



四、常见追问

Q1：param 和 userContext 有什么区别？



答：param 保存页面业务条件，决定本次请求想做什么；userContext 保存调用者身份和权限范围，决定谁在操作以及可以访问哪些数据。



Q2：认证和授权有什么区别？



答：认证用于确认调用者是谁；授权用于判断调用者能执行什么操作、访问哪些数据。



Q3：为什么不能相信前端传入的 storeId？



答：前端请求可以被修改。如果后端直接相信门店 ID，用户可能构造请求访问其他门店数据。真正的数据范围必须由后端根据可信身份确认。



Q4：用户身份和内部服务 token 有什么区别？



答：用户身份表示哪个用户正在操作；内部服务 token 表示 BFF 是否有资格调用内部下游服务。



Q5：为什么前端隐藏按钮不能代替权限校验？



答：前端代码和请求可以被修改，隐藏按钮只是交互控制，真正的安全边界必须由后端执行。



Q6：为什么必须清理 UserContext？



答：云函数实例可能被复用。如果身份上下文残留，可能污染下一次请求，造成日志错误、权限污染甚至越权。



Q7：什么是横向越权？



答：同级用户访问其他用户或其他门店的数据，例如门店 A 员工查看门店 B 的订单。



Q8：什么是纵向越权？



答：普通角色执行更高权限角色才能执行的操作，例如普通客服调用管理员导出或退款接口。



五、容易说错的地方



不要说：



前端传来的 storeId 可以直接作为数据权限条件。



正确说法：



前端参数只能表达查询需求，

数据权限范围必须由后端可信身份决定。



不要说：



OPENID、UNIONID 和内部 token 是同一种凭证。



正确说法：



OPENID 和 UNIONID 用于用户身份识别，

内部 token 用于服务间调用认证。



不要说：



我独立设计了完整的认证和权限平台。



正确说法：



我理解和参与的是 BFF 层身份上下文传递、

数据权限边界和请求结束后的上下文清理。

六、不会答时的安全说法



这个具体权限字段我不想凭记忆乱说。我可以确认的是，业务方法会结合请求参数和用户上下文执行，前端权限参数不能直接作为可信依据。具体角色、门店范围和权限表结构需要以真实用户服务和业务代码为准。
