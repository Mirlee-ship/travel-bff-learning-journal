UserContext

上下文清理

二、核心调用链

前端发起请求

→ 请求进入 biz 统一入口

→ 获取微信或后台登录身份

→ 构造当前请求的 UserContext

→ 读取 moduleName、serviceName、funcName 和 param

→ 调用 service\[funcName](param, userContext)

→ 业务方法结合身份范围执行查询

→ 记录用户和调用日志

→ 返回统一响应

→ finally 中清理 UserContext

三、为什么业务参数和身份信息要分开



页面传入的业务参数可能包括：



statusList

itemName

createdAt

pageIndex

pageSize

tradeOrderId



这些参数只能说明：



用户想查询什么



它们不能证明：



用户是谁

用户属于哪个门店

用户拥有什么权限



因此业务方法还需要接收由系统生成的可信身份上下文。



四、OPENID 和 UNIONID

OPENID



可以理解为：



用户在某一个微信应用中的身份标识

UNIONID



可以理解为：



同一开放平台体系中识别同一个用户的统一标识



具体业务使用哪个字段作为用户主标识，需要以用户服务实现为准。



五、UserContext 的作用



UserContext 可以理解为当前请求的身份档案。



它可能包含：



userId

openId

unionId

employeeId

storeId

role

permissions

requestSource



作用：



入口统一识别一次身份

后续业务方法直接使用

避免每个方法重复解析登录态

统一数据权限和日志身份

六、订单列表权限



订单列表查询通常包含两类条件。



页面条件：



订单状态

商品名称

时间范围

订单号

分页



权限条件：



当前门店

当前员工

允许访问的订单来源

角色对应的数据范围



概念上可以理解为：



查询符合页面条件的订单

并且

订单必须在当前用户的数据权限范围内

七、订单详情权限



详情接口收到：



{

&#x20; "tradeOrderId": "demo-order-001"

}



不能只根据订单 ID 返回数据。



还要判断：



当前用户是否有权查看该订单



概念流程：



查询订单

→ 获取订单所属门店或业务范围

→ 对比当前用户权限范围

→ 有权限则返回

→ 无权限则返回业务错误



也可以在查询条件中直接加入权限范围。



具体方式以真实代码为准。



八、为什么不能相信前端 storeId



前端代码和请求参数可以被修改。



如果前端传入：



storeId = store-B



而后端不校验当前用户真实所属门店，就可能造成横向越权。



安全原则：



页面参数决定查询需求

可信登录态决定数据范围

九、用户身份和内部服务 token



用户身份回答：



哪个用户正在操作？



内部服务 token 回答：



BFF 是否允许调用某个内部服务？



项目入口中从 Redis 读取：



authorizationTokenInside



它属于服务端内部凭证。



内部 token 应该：



只在服务端保存和使用

不返回给前端

不完整打印到普通日志

通过安全配置方式管理

十、前端隐藏按钮不等于安全



前端可以根据权限隐藏导出或修改按钮。



但用户仍可能：



修改前端代码

手动构造请求

直接调用后端接口



因此：



前端隐藏按钮：改善交互

后端权限校验：真正安全边界

十一、横向越权和纵向越权

横向越权



同级用户访问其他用户或门店的数据。



例如：



门店 A 员工查看门店 B 的订单

纵向越权



普通角色执行更高权限角色的操作。



例如：



普通客服执行管理员导出或退款操作

十二、权限校验分层



统一入口层适合处理：



是否登录

token 是否有效

接口是否允许匿名访问

通用身份检查



业务方法层适合处理：



能否查看具体订单

是否属于当前门店

是否允许导出当前范围

当前状态是否允许修改



记忆方式：



入口做通用认证

业务层做具体授权

十三、权限失败



权限不足属于可预期业务错误。



示例：



{

&#x20; "success": false,

&#x20; "message": "无权查看该订单",

&#x20; "errorCode": 40301,

&#x20; "data": null

}



具体错误码以项目实现为准。



普通权限不足不一定需要作为系统故障告警，但高频异常越权行为可以进入安全监控。



十四、日志和身份信息



日志记录用户标识可以帮助排查：



哪个用户发起请求

请求了哪个方法

查询了什么数据

是否执行成功



但日志需要注意：



手机号脱敏

身份证号脱敏

token 不落日志

openId 和 unionId 控制访问权限

限制日志保存周期

十五、为什么清理 UserContext



云函数实例可能复用。



如果身份信息残留，下一次请求可能错误读取上一位用户的信息。



风险包括：



日志身份错误

查询条件错误

权限范围污染

跨用户数据泄漏



所以 finally 中必须清理请求上下文。



十六、职责边界



安全表述：



我理解和参与的是 BFF 层用户上下文传递、

接口身份信息、数据权限边界和请求结束后的清理。



避免夸大为：



我独立设计了公司统一认证和权限平台。

十七、证据边界



当前可以确认项目存在：



userContext 参数传递

OPENID 和 UNIONID 日志上下文

authorizationTokenInside

UserContext.destroy()



具体角色表、权限表和门店校验代码需要以真实源码为准。



十八、Day13 核心结论



UserContext 是当前请求的身份和权限档案。统一入口识别调用者后，将 param + userContext 传给业务方法，由后端根据可信身份控制数据范围。用户身份和内部服务 token 必须分开管理，请求结束后还必须清理上下文。
