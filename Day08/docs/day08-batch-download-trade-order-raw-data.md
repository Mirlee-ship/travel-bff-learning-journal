Day08 学习记录：订单分片导出

一、今天学什么



今天学习后台订单分片导出接口：



trade-bff / web / batchDownloadTradeOrderRawData



它的作用是按照后台订单筛选条件，分批返回导出所需的数据。



二、业务场景



后台订单列表可能符合条件的订单有数千条。



页面为了展示性能，通常只返回当前页：



当前页 20 条

总数 3680 条



但是运营点击导出时，需要导出全部 3680 条，而不是只导出当前页。



因此导出接口需要重新使用列表筛选条件查询完整结果。



三、为什么不能一次性返回全部数据



一次性查询和返回大量订单可能造成：



数据库查询耗时过长

云函数或服务请求超时

内存占用过高

响应体过大

表格生成压力过大

失败后需要重新执行全部任务



因此项目使用分页参数把大结果集拆成多个批次。



四、核心调用链

运营选择订单筛选条件

→ 点击导出

→ 前端保留当前筛选条件

→ 请求 batchDownloadTradeOrderRawData

→ 根据 pageIndex 和 pageSize 查询当前批次

→ 补充和转换导出字段

→ 组装 headerMap 和 data

→ 返回当前批次

→ 继续请求下一批

→ 收集完整结果



RawData 接口可以确认的是按批返回导出数据。最终合并和文件生成位置需要以真实实现为准。



五、核心参数

pageIndex



表示当前获取第几批数据，通常从 0 开始：



pageIndex = 0：第一批

pageIndex = 1：第二批

pageIndex = 2：第三批

pageSize



表示单批最多获取多少条数据。



例如：



pageIndex = 0

pageSize = 100



表示获取第一批 100 条。



categoryList



限定导出订单的商品类目。



一日聚常见：



{

&#x20; "categoryList": \["ONE\_SECOND\_PARTY"]

}

statusList



限定导出的订单状态。



例如：



{

&#x20; "statusList": \["PENDING\_FOR\_SUPPLIER"]

}

时间和其他条件



可能包括：



createdAt

fulfilledAt

itemName

orderNo

contactName

contactMobile

sourceType

门店条件



这些条件应尽量与列表接口保持一致。



六、核心返回



分片返回可以理解为：



{

&#x20; "headerMap": {

&#x20;   "orderNo": "订单号",

&#x20;   "itemName": "商品名称",

&#x20;   "status": "订单状态"

&#x20; },

&#x20; "data": \[

&#x20;   {

&#x20;     "orderNo": "demo-order-001",

&#x20;     "itemName": "一日聚露营套餐",

&#x20;     "status": "待供应商确认"

&#x20;   }

&#x20; ]

}



headerMap 用于描述导出字段和列名。



data 是当前批次的行数据，不代表完整导出结果。



七、列表分页和导出分片的区别



列表分页：



目标：展示当前页

通常由用户手动翻页

返回 list + total



导出分片：



目标：连续取完全部符合条件的数据

通常由导出流程连续拉取

返回 headerMap + data



可以记成：



列表分页：看一页

导出分片：取完全部

八、为什么列表和导出筛选条件要一致



假设页面列表筛选条件是：



一日聚

待供应商确认

1 月份下单

商品名称包含露营



如果导出时漏掉商品名称条件，就会出现：



页面只显示露营订单

导出文件却包含其他商品



因此导出接口应尽量复用列表接口的条件和字段映射。



九、同一次导出的批次规则



同一次导出过程中：



categoryList 不变

statusList 不变

时间范围不变

商品和用户条件不变

pageSize 通常保持不变

pageIndex 逐批递增



如果不同批次使用不同筛选条件，最终数据会混乱。



十、如何判断最后一批



常见方法有：



当前批次条数小于 pageSize



或者：



累计数量达到接口返回的 total



项目具体使用哪种方式，需要以真实协议和代码为准。



十一、失败重试



分片后，如果第 3 批失败，方案上具备只重试第 3 批的条件。



但是否已经实现自动重试，需要以真实代码为准。



安全说法：



分片方案具备按批重试和失败隔离的条件。



不要无证据说：



项目已经实现了完整的自动重试和断点续传。

十二、RawData 和整包导出



项目中还可能存在：



batchDownloadTradeOrder



可以概念性地区分为：



batchDownloadTradeOrderRawData：

按批返回导出表头和数据



batchDownloadTradeOrder：

完成更完整的导出或文件生成流程



具体返回格式和内部实现以代码为准。



十三、BFF 的作用



BFF 在导出链路中可能负责：



接收页面筛选条件

转换成订单查询条件

补充关联字段

转换状态文本

格式化时间

转换金额单位

确定导出列

组装 headerMap 和 data



因此导出不是简单把数据库字段原样返回。



十四、异常和边界

pageSize 过大



可能导致单批请求超时和内存压力，需要合理限制单批大小。



没有数据



可以正常返回：



{

&#x20; "headerMap": {},

&#x20; "data": \[]

}



空结果不是系统异常。



导出期间数据变化



如果导出过程中有新订单产生，不同批次的数据集可能变化。



固定查询时间点、稳定排序和异步快照属于进一步优化方向，不能在没有代码证据时宣称已经实现。



十五、和普通 CRUD 的区别



普通 CRUD 可能只是查询订单并返回。



分片导出还需要处理：



筛选条件复用

大结果集拆分

分页分片

字段映射

表头组装

时间和金额格式化

批次终止判断

失败隔离



更准确的定位是：



面向运营后台的大结果集订单导出接口。

十六、职责边界



安全表述：



我参与和理解的是 BFF 层导出链路，

包括筛选参数映射、分片参数、

返回字段组装和列表导出口径验证。



避免夸大为：



我独立设计了整套文件存储和异步导出平台。

十七、Day08 核心结论



batchDownloadTradeOrderRawData 复用列表筛选条件，通过 pageIndex + pageSize 分批查询完整结果集，并返回 headerMap + data，降低大数据量导出的单次请求风险。
