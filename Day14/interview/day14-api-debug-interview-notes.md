Day14 面试笔记：接口测试与联调

一、30 秒版本



Day14 我复盘的是接口测试和前后端联调。我会先通过 Network 确认页面是否发出请求，再检查 moduleName / serviceName / funcName 和 param 是否正确，然后判断响应属于成功有数据、成功空结果、业务失败还是系统失败。如果请求和响应正常但页面异常，就排查前端字段映射；如果接口系统失败，就根据 traceId 和 requestId 查询后端日志。



二、60 秒版本



联调时我不会只看页面结果，而是沿着完整链路排查。



首先确认页面是否发出请求，然后检查路由字段和业务参数，例如状态枚举、时间单位和分页下标。接着区分成功有数据、成功空结果、业务失败和系统失败四种情况。



对复杂筛选问题，我会构造最小可复现请求，先保留必要参数，再逐个增加条件。系统异常则记录发生时间、funcName、requestId 和 traceId，结合后端结构化日志定位具体下游。



三、90 秒版本



Day14 我重点理解的是接口测试、前后端联调和错误定位。



以订单列表为例，页面点击查询后，我会先通过浏览器 Network 查看请求是否发出，然后确认 trade-bff / web / queryOrderForParty 路由是否正确，并检查 statusList、categoryList、时间范围、pageIndex 和 pageSize。



响应需要区分四类：成功有数据、成功空结果、业务失败和系统失败。空列表不一定是 Bug，可能是筛选条件、分页或权限范围导致；业务失败通常有明确的 message 和 errorCode；系统失败则需要通过 traceId 和 requestId 查询日志。



如果参数复杂，我会构造最小复现，逐个增加筛选条件。联调时使用请求和响应证据沟通，不会直接判断问题归属。我的参与重点是接口参数核对、返回结构验证、问题复现和日志证据整理。



四、常见追问

Q1：联调的标准顺序是什么？



答：



页面操作

→ 是否发出请求

→ 路由字段

→ 业务参数

→ 响应类型

→ 页面渲染

→ traceId 日志

→ 最小复现

→ Bug 记录

Q2：HTTP 200 是否表示业务成功？



答：不一定。HTTP 200 只表示网络请求正常返回，还需要检查业务响应中的 success、data、message 和 errorCode。



Q3：success=true 但 list 为空是否是异常？



答：通常不是。这表示接口执行成功，但当前筛选条件没有匹配数据，需要检查筛选、分页、时间和权限范围。



Q4：如何判断问题偏前端还是后端？



答：没有请求、参数和页面选择不一致，或响应正确但页面不展示，更偏前端；请求正确但响应数据或业务处理错误，更偏后端。必须根据证据判断。



Q5：什么是最小复现？



答：保留最少必要参数稳定重现问题，再逐个加入筛选条件，定位从哪个参数开始出现异常。



Q6：系统异常如何定位？



答：记录接口路由、参数摘要、时间、traceId 和 requestId，再通过后端结构化日志定位具体业务方法或下游服务。



Q7：Bug 应该包含哪些内容？



答：环境、页面、步骤、接口、请求参数、实际结果、预期结果、发生时间、traceId、是否必现和截图。
