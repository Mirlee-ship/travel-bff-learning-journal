\# Day05 学习记录：一日聚订单列表 queryOrderForParty



\## 一、今天学什么



今天学习后台一日聚订单列表接口：



```text

trade-bff / web / queryOrderForParty



它的作用是根据后台页面传入的筛选条件，查询订单列表，并返回当前页数据和总数。



二、业务场景



运营人员打开后台一日聚订单列表时，页面上会有很多筛选条件：



订单状态

商品名称

用户昵称

手机号

订单号

下单时间

出行时间

分页



这些页面动作最终会落到同一个接口：



queryOrderForParty

三、为什么这是 BFF 典型接口



这个接口不是简单查订单表。



因为页面展示一行订单时，不只需要订单本身，还可能需要：



商品信息

用户信息

联系人信息

管家信息

差额信息

子单信息

套餐信息

是否有关联订单

是否合并单



这些信息可能来自不同服务或不同数据结构。



BFF 层的作用是：



接收前端筛选参数

→ 转换成后端查询条件

→ 查询订单

→ 补充关联信息

→ 组装成前端能直接展示的 list

四、核心调用链

后台页面发起订单查询

→ biz 统一入口

→ trade-bff / web / queryOrderForParty

→ 构造查询条件

→ 如果有 itemName，先查商品 ID

→ 如果有 nickName / mobile，先查用户 unionId

→ 查询订单列表

→ 查询订单总数

→ 补充用户、管家、差额、子单等信息

→ 组装页面列表字段

→ 返回 list + total

五、核心入参理解

1\. 分页参数

{

&#x20; "pageIndex": 0,

&#x20; "pageSize": 20

}



含义：



pageIndex：第几页

pageSize：每页多少条



后台订单数据量可能很大，所以必须分页。



2\. 状态筛选

{

&#x20; "statusList": \["PENDING\_FOR\_SUPPLIER"]

}



页面上点击不同订单状态 Tab，本质上就是传不同的 statusList。



3\. 类目筛选

{

&#x20; "categoryList": \["ONE\_SECOND\_PARTY"]

}



一日聚订单列表通常会固定或默认筛选一日聚类目。



4\. 商品名筛选

{

&#x20; "itemName": "露营"

}



商品名不会直接查订单。



更合理的链路是：



itemName

→ 查询商品服务

→ 得到 itemIdList

→ 用 itemIdList 查询订单



原因是订单和商品之间更稳定的关联字段通常是 itemId。



5\. 用户昵称 / 手机号筛选

{

&#x20; "nickName": "张三",

&#x20; "mobile": "13800000000"

}



用户昵称或手机号也不是直接查订单。



更合理的链路是：



nickName / mobile

→ 查询用户服务

→ 得到 unionIdList

→ 用 unionIdList 查询订单



原因是订单里通常保存的是用户唯一标识，而不是直接依赖昵称。



六、核心出参理解



接口最终返回：



{

&#x20; "list": \[],

&#x20; "total": 0

}



list 是当前页订单列表。



total 是符合筛选条件的订单总数。



前端需要 total 来展示分页。



七、为什么要查 list 和 total



列表页不只要知道当前页有什么数据，还要知道总共有多少条。



例如：



当前页显示 20 条

总数 368 条

前端才能计算一共有多少页



所以后端通常要同时查询：



订单列表

订单总数

八、为什么查完订单还要补充信息



订单原始数据里可能只有：



itemId

unionId

privateButlerUnionId

rootOrderId



但页面要展示的是：



商品名称

用户昵称

联系人

手机号

管家名称

差额金额

套餐名称

是否有关联订单

是否合并单



所以 BFF 需要在订单查询之后，批量补充这些页面字段。



九、空结果怎么理解



空结果不是异常。



例如：



商品名查不到商品 ID

用户手机号查不到用户

订单筛选条件太严格



都可以返回：



{

&#x20; "list": \[],

&#x20; "total": 0

}



这表示查询成功，只是没有匹配的数据。



十、Day05 自测题答案

问题 1：queryOrderForParty 是做什么的？它和 Day3 的 searchItem 有什么区别？



queryOrderForParty 是后台一日聚订单列表接口，用来根据状态、商品名、用户信息、时间、分页等条件查询订单列表。



它和 searchItem 的区别是：



searchItem 查商品，面向商品列表页

queryOrderForParty 查订单，面向后台订单列表页



相同点是：



都会把页面输入转换成后端更稳定的 ID 条件

都会分页

都会返回列表和总数

都会处理空结果

问题 2：为什么页面传 itemName 时，不是直接用商品名查订单，而是先查商品 ID？



因为订单和商品之间更稳定的关联字段通常是 itemId。



商品名可能会变化，也可能存在相似名称。

先用商品名查出候选商品 ID，再用 itemIdList 查订单，查询条件更稳定，也更符合订单数据结构。



问题 3：为什么用户昵称、手机号要先转换成 unionIdList？



因为订单里通常保存的是用户唯一标识，例如 unionId。



昵称可能重复，手机号也属于用户信息字段。

所以 BFF 会先通过用户服务查到对应用户，再拿 unionIdList 去查订单。



问题 4：为什么列表接口既要查 list，又要查 total？



list 用来展示当前页数据。

total 用来支持前端分页。



如果只有 list，前端不知道总共有多少条，也不知道一共有多少页。



问题 5：订单列表查出来后，为什么还要补充用户、管家、差额、子单等信息？



因为订单原始数据不一定包含页面展示所需的完整字段。



页面需要展示用户昵称、管家名称、差额金额、是否有关联订单等信息。

这些信息可能来自用户服务、管家关系、差额服务、子单服务，所以 BFF 要补充并组装成页面模型。



问题 6：空结果应该怎么返回？为什么空结果不是异常？



空结果应该返回：



{

&#x20; "list": \[],

&#x20; "total": 0

}



因为没有查到数据是正常业务结果，不是系统错误。

比如商品名没有匹配商品、用户手机号不存在、筛选条件太严格，都属于正常查询结果。
