Day12 学习记录：异步任务、定时任务与 Trigger

一、今天学什么



今天学习后台任务如何被创建、调度和执行。



重点包括：



同步和异步的区别

Trigger 和 Job 的职责

任务状态

分批处理

幂等

分布式锁

失败重试

日志和告警

二、同步任务



同步任务的调用链：



前端发起请求

→ 后端立即执行

→ 前端等待

→ 后端执行完成

→ 返回结果



适合：



订单列表查询

商品详情查询

快速参数校验



特点：



用户需要立即结果

任务通常耗时较短

请求连接会等待业务完成

三、异步任务



异步任务的调用链：



前端发起任务

→ 后端创建任务记录

→ 返回 taskId 和 PENDING

→ 后台 Job 稍后执行

→ 更新状态和结果

→ 前端查询任务状态



适合：



大结果集导出

批量数据处理

批量通知

复杂报表生成

四、定时任务



定时任务不需要用户点击。



调用链：



到达设定时间

→ Trigger 启动任务

→ Job 查询待处理数据

→ 分批执行

→ 记录结果

→ 失败时记录并告警



适合：



清理过期导出文件

扫描超时订单

定时同步数据

生成统计结果



具体项目中有哪些 Trigger，需要以真实配置和源码为准。



五、Trigger 和 Job



Trigger 负责：



什么时候启动任务



Job 负责：



任务启动后执行什么业务



例如导出文件清理：



Trigger：

每天凌晨两点启动



Job：

查询过期文件

→ 删除对象存储文件

→ 更新清理状态

→ 记录日志

六、任务状态



最基础的任务状态：



PENDING

PROCESSING

SUCCESS

FAILED



状态变化可以理解为：



PENDING

→ PROCESSING

→ SUCCESS



失败时：



PENDING

→ PROCESSING

→ FAILED



任务状态是前端查询进度和结果的基础。



七、任务记录



一条任务记录可能包含：



{

&#x20; "taskId": "demo-task-001",

&#x20; "taskType": "ORDER\_EXPORT",

&#x20; "status": "PROCESSING",

&#x20; "progress": 60,

&#x20; "processedCount": 600,

&#x20; "totalCount": 1000,

&#x20; "retryCount": 0,

&#x20; "result": null,

&#x20; "errorMessage": null

}



任务记录不能只保存在单个进程内存里。



原因是：



云函数实例可能重启

系统可能同时运行多个实例

内存数据无法共享

任务可能丢失



因此需要共享存储或任务平台。



八、分批处理



如果需要处理十万条数据，不能一次全部加载到内存。



更合理的方式：



每次处理固定数量

处理完成后继续下一批

直到没有剩余数据



优点：



单次处理量可控

降低内存和超时风险

更容易记录进度

失败时更容易定位批次

九、幂等



幂等表示：



同一个任务执行一次或多次，

最终结果保持一致，

不会重复产生副作用。



需要避免的重复副作用：



重复发送通知

重复扣款

重复生成订单

重复创建导出文件

重复修改业务状态



常见思路：



使用业务唯一键

执行前检查处理记录

成功后写入完成标记

数据库唯一约束

幂等状态表

十、分布式锁



分布式锁用于减少多个实例同时处理同一个任务。



概念流程：



Job 准备处理 taskId

→ 尝试获取分布式锁

→ 获取成功：继续处理

→ 获取失败：说明其他实例正在处理



但锁不能代替幂等。



原因是：



锁可能过期

任务可能超时

平台可能重新触发

执行完成后响应可能丢失



因此更安全的方式是：



锁减少并发重复

幂等保证最终正确

十一、失败重试



临时系统错误可以重试：



网络超时

临时限流

下游短暂不可用

对象存储上传失败



永久业务错误不适合重试：



参数非法

没有权限

数据不存在

业务规则不满足



重试策略需要：



限制最大重试次数

设置下次重试时间

记录 retryCount

保存最后一次错误



不能无限重试。



十二、失败任务隔离



任务超过最大重试次数后，可以：



标记为 FAILED

保存 errorMessage

进入失败任务列表

发送告警

等待人工处理



单个坏任务不能长期阻塞正常任务。



十三、任务进度



长时间任务可以记录：



processedCount

totalCount

progress



计算思路：



progress = processedCount / totalCount × 100%



例如：



{

&#x20; "processedCount": 600,

&#x20; "totalCount": 1000,

&#x20; "progress": 60

}

十四、任务日志



后台 Job 日志可以包含：



taskId

taskType

status

batchIndex

processedCount

successCount

failureCount

retryCount

timeCost

errorMessage

traceId



这样可以定位：



哪个任务失败

哪一批失败

处理了多少数据

失败了多少数据

错误原因是什么

十五、任务告警



后台任务告警可以包含：



taskId

taskType

触发时间

当前批次

参数摘要

重试次数

错误信息

错误堆栈



未知系统异常需要告警。



可预期业务异常通常只记录结果，不需要重复制造告警噪音。



十六、什么时候使用异步任务



适合：



耗时较长

数据量较大

用户不需要立即结果

需要分批处理

需要失败重试

需要定时自动执行



不适合：



任务本身非常快

用户必须立即拿到结果

后续操作依赖当前结果

十七、职责边界



安全表述：



我理解的是异步任务的创建、状态、

Trigger 调度、Job 执行、幂等和重试模型。

十八、证据说明



本日的接口和代码示例主要用于展示后台任务的通用设计模型。



在没有真实 Job 和 Trigger 源码证据前，应说：



这是结合项目场景整理的异步任务学习模型。



不能说：



这些具体接口和状态字段一定存在于生产代码。

十九、Day12 核心结论



异步任务适合大批量、耗时或定时执行的业务。接口可以先创建任务并返回 taskId，再由 Trigger 调度后台 Job 分批执行，执行过程中更新状态、进度和结果，并通过幂等、锁、有限重试、日志和告警保证任务稳定运行。
