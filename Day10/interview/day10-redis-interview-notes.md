Day10 面试笔记：Redis 配置与状态能力

一、30 秒版本



Day10 我复盘的是项目中的 Redis 公共能力。项目通过 vitality-sdk/redis 获取 Redis 实例，例如在 biz/index.ts 中使用 hget 从 global:feature:config 读取 tokenKey，用于高频配置读取。Redis 还可以承接短期状态标记和轻量队列，但关键订单、支付和库存数据仍然要以正式数据库或领域服务为准。



二、60 秒版本



在这个项目中，Redis 更偏公共支撑能力，而不是订单的最终存储。



一个明确的例子是在 biz 统一入口中获取 Redis 实例，然后通过 hget 读取全局配置中的 tokenKey，写入当前进程环境，供后续业务调用使用。



Redis 适合高频配置、短期状态和轻量队列。使用时需要考虑 TTL、缓存更新、回源和服务不可用场景。关键订单、支付和库存不能只依赖 Redis。



三、90 秒版本



Day10 我重点理解的是 Redis 在 BFF 项目中的定位。项目通过 vitality-sdk/redis 获取统一客户端，在 biz/index.ts 中使用 hget 从 global:feature:config 读取 tokenKey，然后放入当前进程环境变量中，供后续下游调用使用。



这个场景说明 Redis 主要承担高频配置读取，减少每次请求访问远程配置来源的成本。



Redis 还常用于短期状态和轻量队列，例如通过 setex 保存带 TTL 的状态，通过 lpop 取出队列元素。不过这些具体业务只有看到真实代码时才能说成项目已实现。



缓存不能当作订单、支付或库存的唯一真相。Redis 需要考虑 TTL、旧数据、回源和不可用场景。我的理解和参与重点是客户端调用、配置读取和缓存边界，不是 Redis 集群和底层运维。



四、常见追问

Q1：Redis 在项目中做什么？



答：当前明确证据是用于高频全局配置读取；它也可以用于短期状态和轻量队列，减少重复查询和跨服务调用。



Q2：hget 是什么意思？



答：hget 用来读取 Redis Hash 中某一个字段。项目中从 global:feature:config 读取 tokenKey。



Q3：什么是 TTL？



答：TTL 是 key 的剩余生存时间。到期后 Redis 自动删除数据，用于控制缓存和临时状态的生命周期。



Q4：为什么不能把 Redis 当成订单唯一真相？



答：Redis 数据可能过期、丢失或与正式数据短期不一致。关键订单和支付数据应以领域服务或数据库为准。



Q5：缓存没有命中怎么办？



答：业务允许时可以回源数据库或下游服务，再根据需要写回 Redis。关键配置要根据安全要求决定回源还是直接失败。



Q6：Redis 不可用怎么办？



答：普通缓存可以考虑回源或降级，关键鉴权配置不能使用不安全默认值，需要记录错误并按业务规则处理。



Q7：setex 和 set 有什么区别？



答：set 写入普通值，setex 在写入值的同时设置过期时间，更适合短期状态和缓存。



Q8：Redis List 能代替消息队列吗？



答：Redis List 可以承接轻量队列，但强可靠、复杂重试和消息确认场景通常需要专业消息队列方案。

正确说法：



Redis 是缓存和状态支撑层，

关键数据以正式领域服务或数据库为准。



不要说：



我独立设计了 Redis 集群和缓存高可用架构。
