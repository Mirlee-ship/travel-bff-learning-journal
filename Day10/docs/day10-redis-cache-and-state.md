Day10 学习记录：Redis 配置缓存与状态能力

一、今天学什么



今天学习 Redis 在 BFF 项目中的作用。



重点不是 Redis 底层实现，而是：



项目为什么使用 Redis

代码如何读取 Redis

TTL 有什么作用

Redis 有哪些风险和边界

二、项目中的代码证据



统一入口中存在：



const redis = getRedisInstance()



process.env.authorizationTokenInside = await redis.hget(

&#x20; 'global:feature:config',

&#x20; 'tokenKey',

)



对应调用链：



请求进入 biz 云函数

→ 获取 Redis 客户端

→ 从 Hash 中读取 tokenKey

→ 写入当前进程环境变量

→ 后续业务使用配置

三、为什么用 Redis 读取配置



如果每次请求都访问数据库或远程配置服务：



调用链更长

响应时间增加

下游压力增加

配置服务抖动可能影响业务



使用 Redis 后：



请求进入

→ 直接读取 Redis

→ 获取配置

→ 继续业务



可以降低重复访问成本。



四、Redis 和正式数据库的区别

对比项	Redis	正式数据库或领域服务

主要用途	缓存、状态、队列	正式业务数据

读取速度	通常较快	相对较慢

生命周期	可能过期或被清理	长期保存

一致性	可能短期不一致	通常作为真实来源

示例	配置、标记、临时状态	订单、支付、商品



一句话记忆：



数据库保存真相，Redis 提高速度。

五、Redis Hash



代码：



redis.hget('global:feature:config', 'tokenKey')



可以理解为从下面的配置对象中读取一个字段：



{

&#x20; "tokenKey": "demo-value",

&#x20; "switchA": "on",

&#x20; "switchB": "off"

}



使用 Hash 的好处：



相关配置可以放在同一个 key 下

可以只读取某个字段

配置结构更清晰

六、TTL 和 setex



setex 可以理解为：



set + expire



模拟场景：



写入某个任务状态

同时设置 10 分钟后过期



适合：



临时状态

验证码

防重复标记

短期缓存



TTL 需要根据：



数据变化频率

业务可接受的不一致时间

回源成本

安全要求



进行设置。



七、缓存命中与缓存未命中



缓存命中：



Redis 有数据

→ 直接使用



缓存未命中：



Redis 没有数据

→ 查询正式数据源

→ 必要时写回 Redis

→ 返回结果



不是所有数据都必须回源。



关键鉴权配置读取失败时，不能随便使用不安全的默认值。



八、状态标记



Redis 可以保存短期状态，例如：



某个任务是否正在处理

某个通知是否已经发送

某个操作是否处于防重复时间内



这类状态通常：



读取频繁

生命周期较短

不适合作为永久正式记录



具体项目业务是否使用，需要以代码证据为准。



九、轻量队列



Redis List 可以保存待处理元素。



例如：



任务 A

任务 B

任务 C



执行 lpop 后：



取出任务 A

队列剩下任务 B 和任务 C



Redis List 可以承接轻量队列，但不能直接等同于专业消息队列平台。



十、Redis 的风险



常见风险：



缓存和真实数据不一致

旧数据没有及时过期

大量 key 同时失效

热点 key 压力过大

Redis 不可用

把缓存误当作唯一数据源

十一、缓存穿透、击穿和雪崩

缓存穿透

Redis 没有

数据库也没有

每次请求都访问数据库

缓存击穿

一个热点 key 突然过期

大量请求同时回源

缓存雪崩

大量 key 同时过期

或 Redis 整体不可用

大量请求同时访问下游



这些属于 Redis 常见风险。



本项目中是否实现复杂治理方案，需要以真实代码为准。



十二、Redis 不可用时怎么办



需要根据数据重要性判断。



普通缓存：



可以尝试回源

可以使用合理默认值



鉴权或安全配置：



不能使用不安全默认值

可能需要明确失败

记录日志并告警



所以不能统一地说 Redis 失败全部忽略，也不能说所有场景都必须中断。



十三、为什么不能缓存所有数据



不适合缓存的场景：



强一致性要求高

数据变化非常频繁

查询成本本身很低

缓存命中率很低

每次查询参数都不同



例如支付状态和库存状态，需要谨慎处理缓存时间。



十四、职责边界



安全表述：



我在项目中接触和理解的是 Redis 客户端调用、

全局配置读取、TTL 和状态标记等能力。

Redis 集群、高可用和底层运维不是我负责的。



避免夸大为：



我独立设计了 Redis 集群和缓存架构。

十五、Day10 核心结论



Redis 在项目中主要用于高频配置读取、短期状态和轻量队列。已经确认的代码是在 biz/index.ts 中通过 hget 读取 global:feature:config 里的 tokenKey。关键业务数据仍然要以正式数据库或领域服务为准。
