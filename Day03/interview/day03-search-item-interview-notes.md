\# Day 3：商品检索接口面试笔记



\## 一、项目亮点名称



旅行社商品搜索、筛选、排序与分页统一接口。



\---



\## 二、一句话介绍



`searchItem` 使用统一参数模型承接关键词、价格、行程天数、供给来源、排序和分页，并在查询完成后批量补充商品头图和关联城市，最终返回稳定的 `data + total`。



\---



\## 三、30 秒面试话术



旅行社商品页使用统一的 `searchItem` 接口承接关键词、价格区间、行程天数、供给来源、排序和分页。关键词和标签会先转换成候选商品 ID，随后服务动态构造参数化 SQL。列表和总数通过 `Promise.all` 并行查询，再批量补充商品头图和关联城市，最终组装成统一的 `data + total` 返回前端。



\---



\## 四、90 秒面试话术



我参与和重点复盘的是旅行社商品检索接口 `searchItem`。这个接口没有把关键词、筛选和排序拆成多套协议，而是通过统一参数模型承接不同查询场景。



查询开始时，如果传入关键词或标签，会先调用已有搜索能力或标签关系查询，将其转换成候选商品 ID。如果候选集为空，就提前返回空结果，避免继续执行无意义查询。



接下来服务根据零售价、结算价、行程天数和供给来源等参数动态追加 SQL 条件。用户输入通过占位符和参数数组传递，不直接拼接进 SQL。排序支持零售价、结算价和利润的升降序，未传排序时默认按零售价降序。



列表查询和总数查询互不依赖，因此通过 `Promise.all` 并行执行。拿到当前页商品后，再批量查询头图和关联城市，避免逐条调用造成 N+1 查询。最后解析标签、计算利润并返回 `data + total`。



\---



\## 五、完整调用链



```text

商品页面提交搜索条件

→ searchItem

→ 关键词或标签转换成候选商品 ID

→ 动态组装 conditions 和 queryParams

→ 构建列表 SQL 和总数 SQL

→ 添加排序与分页

→ Promise.all 并行查询

→ 批量补充头图与城市

→ 解析标签并计算利润

→ 返回 data + total

```



\---



\## 六、为什么使用统一接口



关键词、价格、天数、来源和排序都属于同一个商品检索模型。



统一接口的收益：



\- 前端只维护一套调用协议

\- 筛选条件可以自由组合

\- 分页返回结构保持一致

\- 新增筛选字段时更容易扩展

\- 联调和回归测试路径更集中



\---



\## 七、为什么关键词先转换成商品 ID



关键词搜索先复用已有 `commonSearchItem` 能力，得到候选商品 ID。



之后在候选商品范围内继续执行价格、天数和来源筛选。



这使文本搜索与结构化筛选职责分离。



\---



\## 八、为什么使用参数化 SQL



```text

conditions：

保存 SQL 条件模板



queryParams：

保存真实参数值

```



使用占位符可以让查询结构与用户输入分离，降低 SQL 注入风险。



\---



\## 九、为什么查询列表和总数



列表用于展示当前页商品。



总数用于计算分页。



两者使用相同的筛选条件，但列表包含排序和分页，总数不包含分页。



\---



\## 十、为什么使用 Promise.all



列表查询和总数查询相互独立，可以同时执行。



`Promise.all` 可以减少总体等待时间，但仍然是两次数据库查询。



\---



\## 十一、为什么批量查询头图和城市



如果对当前页每个商品分别查询关联信息，会产生大量数据库或服务调用。



项目先收集所有商品 ID，再批量查询头图和城市，并根据商品 ID 合并结果，从而避免 N+1 查询问题。



\---



\## 十二、空结果为什么不是异常



没有匹配商品属于正常业务结果。



接口仍然返回：



```json

{

&#x20; "success": true,

&#x20; "data": {

&#x20;   "data": \[],

&#x20;   "total": 0

&#x20; }

}

```



系统异常才会返回 `success=false`。



\---



\## 十三、与普通 CRUD 的区别



普通 CRUD 可能只是从一张表查询商品。



该接口还需要：



\- 搜索候选集转换

\- 动态条件组合

\- 多字段业务排序

\- 列表和总数并行查询

\- 关联信息批量补充

\- 标签格式转换

\- 利润计算

\- 页面模型组装



因此更接近商品检索与数据聚合服务。



\---



\## 十四、潜在优化点



\### 1. 候选商品数量限制



关键词搜索一次性获取固定数量候选商品，命中结果非常多时可能被截断。



\### 2. IN 条件长度



候选商品 ID 过多时，SQL 的 IN 条件会变长。



\### 3. 分页参数校验



需要校验 pageIndex、pageSize 的类型、非负值和最大值。



\### 4. 标签与关键词组合语义



标签和关键词同时存在时，应明确使用交集、并集还是覆盖关系。



\### 5. 利润口径



SQL 排序使用原始利润，但页面可能把负利润显示为 0，需要明确排序和展示口径。

\## 十六、高频追问



\### Q1：为什么不用多个搜索接口？



因为这些条件都属于同一个商品检索模型。统一协议更容易组合条件，也能保持分页和返回结构一致。



\### Q2：怎么证明排序是后端完成的？



保持其他参数不变，只修改 `sortType`，对比响应体中的商品顺序。如果响应顺序变化，说明排序在后端查询中生效。



\### Q3：为什么不是直接使用 LIKE？



项目已经存在统一搜索服务，关键词会先转换成候选商品 ID，再进入结构化筛选。这样可以复用已有搜索能力。



\### Q4：为什么不把头图直接放进主查询？



当前主查询使用检索数据结构，头图和城市来自其他数据源。服务使用批量查询补充，避免逐条查询。



\### Q5：Promise.all 是否只执行了一次 SQL？



不是。列表 SQL 和总数 SQL 仍然是两次查询，只是同时开始执行。



\### Q6：空列表为什么还是 success=true？



因为查询正常完成，只是没有符合条件的商品。空结果是业务结果，不是系统异常。



\### Q7：这个接口最大的技术价值是什么？



将文本搜索、结构化筛选、业务排序、分页、并行查询和关联数据聚合收敛在一条稳定检索链路中。
