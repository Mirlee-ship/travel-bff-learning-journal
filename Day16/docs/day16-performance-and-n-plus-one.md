Day16 学习记录：性能优化、N+1 与批量处理

一、性能问题通常来自哪里



接口变慢通常不是单独某一行代码的问题，而可能来自：



数据库查询次数过多

下游服务调用次数过多

多个任务完全串行

返回数据量过大

一次加载大量数据

缺少分页

字段查询过多

数据库索引不合理

并发数量失控

缓存没有命中

二、N+1 查询



示例流程：



查询 20 条订单

→ 遍历订单

→ 每条订单查询一次商品



调用次数：



订单列表查询：1 次

商品查询：20 次

总调用次数：21 次



如果每条订单还要查询用户和门店：



订单查询：1 次

商品查询：20 次

用户查询：20 次

门店查询：20 次

总调用次数：61 次



N+1 的问题会随着列表数量增长。



三、N+1 的代码特征



看到下面结构时要警惕：



for (const order of orders) {

&#x20; const item = await queryItemById(order.itemId)

}



问题不在于循环本身，而在于：



循环内部执行数据库或远程服务调用

四、批量查询方案



优化流程：



1\. 查询订单列表

2\. 收集所有 itemId

3\. 使用 Set 去重

4\. 批量查询商品

5\. 转换为 Map

6\. 根据 itemId 回填商品



示例：



const itemIds = \[

&#x20; ...new Set(

&#x20;   orders

&#x20;     .map(order => order.itemId)

&#x20;     .filter(Boolean),

&#x20; ),

]



批量查询：



const items = await batchGetItems(itemIds)



构造 Map：



const itemMap = new Map(

&#x20; items.map(item => \[item.id, item]),

)



回填：



const result = orders.map(order => ({

&#x20; ...order,

&#x20; itemInfo: itemMap.get(order.itemId) ?? null,

}))

五、为什么要去重



假设 20 条订单属于同一个商品。



未去重：



\[item-1, item-1, item-1, item-1 ...]



去重后：



\[item-1]



去重可以减少：



请求体大小

查询条件数量

数据库处理量

下游重复工作

六、为什么使用 Map



如果每条订单都执行：



items.find(item => item.id === order.itemId)



每次都要遍历数组。



转换为 Map 后：



itemMap.get(order.itemId)



可以直接根据 ID 获取关联数据，更适合批量组装。



七、串行和并行

串行

const item = await getItem()

const user = await getUser()

const store = await getStore()



如果每个任务耗时 100ms，总耗时可能接近：



300ms

并行

const \[item, user, store] = await Promise.all(\[

&#x20; getItem(),

&#x20; getUser(),

&#x20; getStore(),

])



如果三个任务互不依赖，总耗时可能接近最慢的一项。



八、Promise.all 的边界



危险示例：



await Promise.all(

&#x20; orders.map(order => queryItemById(order.itemId)),

)



如果订单有 10000 条，就可能瞬间发出 10000 个请求。



风险：



数据库连接耗尽

下游触发限流

内存占用上升

大量请求同时超时

服务雪崩



所以：



Promise.all 不是查询次数优化

Promise.all 也不是无限并发工具

九、受控并发



当下游不支持批量查询时，可以限制同时执行的任务数量。



例如：



100 个任务

最大并发 10



目标：



比完全串行快

又不让下游瞬间承受 100 个请求

十、分页



订单列表如果一次返回全部数据，会造成：



数据库查询慢

响应体过大

服务内存占用高

前端渲染困难



分页后：



每次只查询 20 或 50 条



常见返回：



list：当前页

total：符合条件的总数

十一、total 查询也可能慢



复杂 COUNT 查询可能包含：



多表关联

模糊搜索

状态条件

时间条件

权限条件



列表只返回 20 条可能很快，但统计全部匹配数量仍然可能较慢。



优化方向：



合理索引

减少不必要关联

汇总表

统计缓存

根据产品需求调整 total 精度

十二、字段裁剪



列表页通常只需要：



订单号

商品名称

状态

金额

联系人

创建时间



列表查询不应该无条件返回全部详情字段。



字段裁剪可以降低：



数据库读取量

网络传输量

服务内存

前端处理成本

日志体积

十三、按需查询



并不是每一笔订单都需要：



拼团信息

旅游出行人

退款信息

履约信息



可以根据：



订单类型

商品类型

订单状态

页面模块



决定是否查询对应信息。



不查询不会使用的数据，往往比优化查询本身更有效。



十四、导出性能



导出比列表风险更高，因为需要处理完整结果集。



风险包括：



全量查询超时

内存占用过高

逐条补充关联字段

Excel 文件过大

对象存储上传失败



优化方式：



分片查询

批量补充关联信息

限制并发

流式或分批写文件

异步任务

失败重试

十五、数据库索引



索引适合高频查询和排序条件，例如：



orderNo

status

createdAt

storeId

itemId



但不能盲目给所有字段加索引。



索引成本：



占用存储

写操作变慢

维护成本增加

组合索引顺序影响查询



正确思路：



结合真实 SQL

检查执行计划

查看扫描行数

确认索引是否命中

十六、Redis 缓存



Redis 适合：



高频读取

变化不频繁

允许短期不一致

查询成本较高



但如果问题本身是 N+1 或不分页，直接加缓存只是暂时掩盖问题。



优化顺序：



先修正查询结构

再考虑缓存

十七、通过日志定位性能问题



入口总耗时只能说明接口慢。



更有价值的是记录阶段耗时：



订单查询耗时

商品查询耗时

用户查询耗时

下游服务耗时

结果组装耗时



还可以记录：



数据条数

批量 ID 数量

数据库调用次数

缓存命中情况

下游调用次数

十八、性能优化流程

记录优化前数据

→ 找到最慢阶段

→ 确认重复调用

→ 设计批量或分页方案

→ 控制并发

→ 修改代码

→ 验证结果正确

→ 对比优化后数据



不能只说：



感觉这里比较慢

所以改成 Promise.all

十九、职责边界



安全表达：



我理解并能识别列表和导出链路中的 N+1、

分页、批量查询和并发控制问题。



避免说：



我主导了项目完整性能架构改造。

二十、核心结论

先减少查询和调用次数

再减少数据量

然后合理控制并发

最后通过数据验证优化结果
