一句话理解：



性能优化不是盲目增加并发，

而是先减少调用次数和数据量，

再合理控制并发和等待时间。

二、和项目链路的关系



项目中的以下接口都可能遇到性能问题：



订单列表

订单详情

订单导出

订单汇总

商品搜索



例如：



订单列表查询 20 条订单

→ 每条订单单独查询一次商品

→ 产生 1 + 20 次查询



这就是典型的 N+1 查询。



三、什么是 N+1



N+1 表示：



先查询 N 条主数据

再为每条主数据单独查询关联数据



例如：



查询订单列表：1 次

查询 20 条订单对应的商品：20 次

总调用次数：21 次



数据量增大后，调用次数也会线性增长。



四、如何解决 N+1



标准处理流程：



查询主数据

→ 收集关联 ID

→ 对 ID 去重

→ 批量查询关联数据

→ 转换成 Map

→ 回填到主数据



调用次数从：



1 + N



降低为：



1 + 1

五、Promise.all 能否解决 N+1



不能完全解决。



Promise.all：

把 N 次查询从串行改成并行



批量查询：

直接减少查询次数



例如 1000 条数据：



Promise.all 仍然会发出 1000 次请求

批量查询可能只需要 1 次或少量分批请求



因此优先级通常是：



先减少调用次数

再合理并发

六、受控并发



如果下游没有批量接口，可以使用有限并发。



例如：



总任务数：100

最大并发数：10



这样可以：



比完全串行更快

避免一次发出 100 个请求

降低数据库或下游服务压力

七、列表、详情和导出的优化重点

列表接口

分页

字段裁剪

批量查询关联信息

避免 N+1

合理索引

详情接口

先查询基础信息

根据订单类型按需查询

互不依赖的模块合理并行

不需要的模块不查询

导出接口

分片查询

批量补充关联数据

控制并发

控制内存

避免一次加载全部数据

八、Redis 的作用



Redis 适合缓存：



读取频率高

变化不频繁

查询成本较高

允许短期不一致



例如：



全局配置

功能开关

短期状态

热点基础数据



但 Redis 不能掩盖：



N+1

不分页

返回字段过多

无限并发



应该先修正不合理查询，再考虑缓存。



九、性能优化标准步骤

1\. 记录优化前基线

2\. 根据 timeCost 和日志找到瓶颈

3\. 统计数据库和下游调用次数

4\. 检查是否存在重复查询

5\. 优先使用批量查询

6\. 减少不必要字段和数据量

7\. 合理设置分页和并发

8\. 再次测量优化结果

9\. 验证业务结果没有变化

十、Day16 必须掌握的 10 句话

N+1 是查询主数据后，再逐条查询关联信息。

N+1 的调用次数会随着数据量增长。

批量查询前应该先提取并去重关联 ID。

批量查询结果可以转成 Map 再回填。

Promise.all 只改变等待方式，不减少查询次数。

无限并发可能压垮数据库或下游服务。

没有批量接口时可以使用受控并发。

列表需要分页和字段裁剪。

大数据导出需要分片和并发控制。

性能优化必须先测量，再修改，最后验证。
