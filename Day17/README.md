Day17：数据一致性、事务、幂等与失败补偿

一、当天核心目标



Day17 学习一次业务操作包含多个步骤时，如何避免出现：



一部分成功

一部分失败

重复执行

并发覆盖

状态不一致



核心内容：



数据一致性

本地事务

跨服务一致性

幂等

有限重试

失败补偿

最终一致性

并发控制

Outbox 思想



一句话理解：



事务解决同一个数据库内的原子性，

幂等解决重复执行，

重试和补偿解决跨服务失败后的恢复。

二、为什么会出现数据不一致



例如确认订单时需要：



更新订单状态

写状态日志

创建履约记录

发送通知



如果结果是：



订单状态更新成功

履约记录创建失败



页面可能显示订单已确认，但系统中没有对应履约记录。



这就是数据不一致。



因此多步骤业务不仅要考虑正常执行，还要考虑：



中间步骤失败怎么办

重复请求怎么办

并发修改怎么办

跨服务部分成功怎么办

三、本地事务



本地事务适用于：



同一个数据库

同一个事务连接

需要一起提交的多步操作



例如：



更新订单表

写订单状态日志

创建履约记录



事务目标：



全部成功后提交

任意一步失败则全部回滚



本地事务不能直接控制：



支付服务

短信服务

对象存储

第三方系统

其他独立数据库

四、事务 ACID

Atomicity 原子性

一组操作要么全部成功，要么全部失败。

Consistency 一致性

事务执行前后，数据仍满足业务规则。

Isolation 隔离性

多个事务并发执行时，尽量减少相互干扰。

Durability 持久性

事务提交后，结果应被可靠保存。

五、幂等



幂等表示：



同一个业务请求执行一次或多次，

最终业务结果保持一致。



例如退款请求第一次已经成功，但响应途中网络中断。



客户端再次提交时，系统不能再次退款。



常见幂等方案：



业务唯一键

幂等请求号

数据库唯一约束

状态校验

事件处理记录

任务结果复用

六、为什么重试必须配合幂等



调用方看到失败，不一定代表服务端没有执行成功。



例如：



服务端已经创建任务

→ 返回响应时网络断开

→ 调用方认为失败

→ 再次发起请求



如果没有幂等，可能创建两个相同任务。



因此：



只要系统支持重试，

就必须考虑重复请求。

七、重试



适合重试：



网络超时

服务临时不可用

对象存储偶发失败

临时限流



不适合重试：



参数错误

权限不足

订单不存在

业务条件不满足

非法状态流转



重试需要：



最大次数

重试间隔

错误类型判断

幂等保护

最终失败状态

八、补偿



补偿是：



前面的操作已经提交，

后续步骤失败时，

通过新的反向业务操作修正结果。



例如：



订单创建成功

库存已经锁定

支付失败



补偿可能是：



关闭订单

释放库存



回滚与补偿的区别：



回滚：

事务提交前撤销数据库修改



补偿：

事务已经提交后，通过新的业务操作修正



补偿本身也必须具备幂等性。



九、最终一致性



最终一致表示：



短时间内允许多个系统状态不同，

但经过消息、重试或补偿后，

最终达到一致。



适合最终一致的场景：



通知

积分

报表

搜索索引

统计数据



核心资金和库存业务需要更谨慎。



十、并发一致性



例如两个请求同时处理一笔订单：



请求 A：确认订单

请求 B：取消订单



两边都读取到：



status = PENDING



如果直接更新，后写入的请求可能覆盖前面的结果。



常见控制方式：



带旧状态的条件更新

version 乐观锁

数据库事务

分布式锁

业务幂等



示例：



UPDATE trade\_order

SET status = 'CONFIRMED'

WHERE id = ?

AND status = 'PENDING'



如果影响行数为 0，说明状态已被其他请求修改。



十一、Outbox 思想



问题：



订单事务提交成功

但业务事件发送失败



Outbox 思路：



同一个数据库事务中：

1\. 更新订单

2\. 写一条待发送事件记录



事务提交后：



后台任务扫描事件表

→ 发送消息

→ 成功后标记已发送

→ 失败则重试



这样可以保证：



只要业务事务提交成功，

待发送事件记录就一定存在。



这是通用设计思想，不代表项目中已经确认使用。



十二、Day17 必须掌握的 10 句话

数据一致性表示同一业务事实在不同表和服务中相互协调。

本地事务只能保证同一个数据库边界内的原子性。

跨服务操作不能依赖一个数据库事务统一回滚。

幂等表示同一请求重复执行，最终结果保持一致。

重试必须配合幂等，否则可能产生重复业务结果。

临时系统错误可以有限重试，永久业务错误不应重复重试。

回滚发生在事务内部，补偿用于修正已经提交的结果。

非核心副作用可以通过异步事件实现最终一致。

并发更新可以通过旧状态条件或 version 控制。

补偿和消息消费本身也需要幂等。
