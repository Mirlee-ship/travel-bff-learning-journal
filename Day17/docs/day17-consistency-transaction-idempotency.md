Day17 学习记录：一致性、事务、幂等与补偿

一、多步骤业务为什么危险



一项业务操作可能同时涉及：



更新数据库

创建关联记录

调用下游服务

发送消息

写入对象存储

更新任务状态



只要步骤超过一个，就可能出现：



第一步成功

第二步成功

第三步失败



系统需要提前考虑：



前面已经成功的操作怎么办

是否可以回滚

是否需要重试

是否需要补偿

重复请求如何处理

二、订单确认示例



假设订单确认需要：



1\. 更新订单状态为 CONFIRMED

2\. 写确认时间

3\. 创建履约记录

4\. 写状态流转日志

5\. 发送通知



如果步骤 1 成功，步骤 3 失败，可能出现：



订单显示已确认

但没有履约记录



因此应该划分：



核心本地事务

更新订单状态

创建履约记录

写状态日志



如果它们在同一个数据库，可以放入事务。



外部副作用

发送通知

更新搜索索引

更新统计数据



可以在事务提交后异步处理。



三、本地事务边界



本地事务适用于：



同一数据库

同一事务连接

同一批原子操作



概念代码：



await database.transaction(async transaction => {

&#x20; await updateOrderStatus(orderId, transaction)

&#x20; await createFulfillment(orderId, transaction)

&#x20; await createStatusLog(orderId, transaction)

})



其中任何一步失败：



事务整体回滚

四、为什么事务中不建议执行长时间远程调用



危险流程：



开启事务

→ 更新订单

→ 调用外部服务等待数秒

→ 提交事务



风险：



数据库连接占用时间长

锁持有时间长

并发能力下降

死锁风险增加



更常见的设计是：



事务内只完成核心数据库操作

→ 尽快提交

→ 事务外处理远程副作用



具体是否允许异步，需要根据业务一致性要求判断。



五、跨服务一致性



订单、支付、履约和通知通常是独立服务。



例如退款流程：



订单服务更新退款状态

→ 支付服务执行真实退款

→ 履约服务关闭履约

→ 消息服务通知用户



一个数据库事务不能同时控制所有服务。



常见方案是：



本地事务

\+

业务事件

\+

幂等

\+

有限重试

\+

失败补偿

六、强一致和最终一致

强一致

操作完成后，

所有相关位置立即看到一致结果。



适合：



核心金额

核心订单状态

库存扣减

最终一致

短时间允许状态不同，

经过消息和重试后最终一致。



适合：



通知

积分

报表

搜索索引

统计



选择依据：



业务风险

用户能否接受延迟

失败恢复成本

系统可用性要求

七、幂等请求



幂等请求通常包含：



业务唯一标识

或者

idempotencyKey



处理流程：



收到请求

→ 检查幂等记录

→ 已成功则返回原结果

→ 未处理则开始执行

→ 保存处理结果

→ 返回结果



状态可以设计为：



PROCESSING

SUCCESS

FAILED



需要注意：



FAILED 是否允许重试

需要根据具体错误类型判断

九、状态校验为什么不一定够



第一次处理：



订单状态更新成功

通知发送失败



第二次请求发现订单已经是新状态。



如果直接返回成功：



通知可能永远没有发送



因此复杂业务可能需要分别记录：



状态是否更新

通知是否完成

履约是否创建

事件是否消费

十、重试设计



重试必须判断错误类型。



适合重试：



连接超时

网络抖动

下游临时不可用

临时限流



不适合重试：



参数错误

权限不足

资源不存在

业务规则不允许



重试策略需要：



最大次数

退避时间

幂等保护

错误记录

最终失败状态



示例：



第一次失败：1 秒后

第二次失败：5 秒后

第三次失败：30 秒后

超过上限：标记 FAILED

十一、补偿



补偿用于已经部分成功的跨服务操作。



示例一：



订单创建成功

库存锁定成功

支付永久失败



补偿：



关闭订单

释放库存



示例二：



文件上传成功

数据库记录保存失败



补偿：



删除孤儿文件

或者由后台任务定期清理

十二、补偿为什么也需要幂等



补偿任务可能重复执行。



例如释放库存执行两次，可能导致库存错误增加。



正确做法应该是：



检查库存锁定记录状态

只允许 LOCKED 变为 RELEASED



而不是简单执行：



库存数量加一

十三、并发更新



两个请求同时处理同一订单时，可能发生覆盖。



解决方式：



带旧状态条件

SET status = 'CONFIRMED'

WHERE id = ?

AND status = 'PENDING'

version 乐观锁

UPDATE trade\_order

SET status = ?,

&#x20;   version = version + 1

WHERE id = ?

AND version = ?

分布式锁

对同一业务 ID 串行处理



分布式锁仍建议配合：



状态校验

条件更新

幂等

十四、缓存一致性



问题：



数据库更新成功

Redis 仍然保存旧数据



常见思路：



先更新数据库

再删除缓存

设置合理 TTL

缓存未命中时回源数据库



缓存通常允许短期不一致。



正式业务事实仍以：



数据库

或者

领域服务



为准。



十五、导出链路中的一致性



导出流程：



查询数据

→ 生成文件

→ 上传对象存储

→ 保存文件记录

→ 返回下载地址



可能出现：



文件生成成功，上传失败

上传成功，记录保存失败

记录成功，返回响应超时



处理方式可能包括：



失败重试

删除孤儿文件

任务幂等

复用已有成功结果

定期清理异常资源

十六、异步任务中的一致性



任务状态：



PENDING

PROCESSING

SUCCESS

FAILED



危险情况：



任务实际执行成功

但 SUCCESS 状态更新失败



调度平台可能再次执行任务。



所以异步任务需要：



taskId

幂等检查

结果记录

重试次数

失败原因

十七、Outbox 思想



业务事务中同时写入：



业务数据

待发送事件记录



事务提交后由后台任务：



读取待发送事件

→ 发送消息

→ 标记发送成功



优势：



业务数据提交成功时，

事件记录不会丢失。



消费者还需要通过 eventId 实现消费幂等。



十八、一致性问题排查步骤

1\. 明确业务最终目标

2\. 找出涉及的表和服务

3\. 查看每一步是否成功

4\. 检查事务是否提交或回滚

5\. 检查是否存在重复请求

6\. 检查重试是否发生

7\. 检查补偿是否执行

8\. 对比各系统最终状态

十九、职责边界



安全表达：



我理解多步骤业务的一致性风险，

能够区分事务、幂等、重试和补偿的职责。



避免说：



我独立实现了完整分布式事务方案。

二十、核心结论

事务负责同库原子性

幂等负责重复执行

重试负责临时失败

补偿负责部分成功

事件负责最终一致

日志负责全过程追踪
